### 修改前
- **Heartbeat 频率**: 默认（约 30 分钟 1 次）

- **优势**: 
  - 响应更及时
  - 减少卡顿和死机
  - 更好的任务连续性

## 📊 心跳频率对比

| 频率 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **1 分钟** | 响应快、不卡顿 | Token 消耗略高 | 活跃开发期、需要快速响应 |
| 5 分钟 | 平衡 | 适中 | 日常使用 |
| 30 分钟 | 节省 Token | 可能卡顿、死机 | 低频使用、后台运行 |

---

## 💡 最佳实践

### 1 分钟心跳的使用建议

**✅ 适合**:
- 开发调试阶段
- 需要快速响应的任务
- 多任务并行处理
- 避免系统卡顿

**⚠️ 注意**:
- Token 消耗会增加
- 如果长时间无交互，可以临时调低频率
- 可以在 HEARTBEAT.md 中添加简单检查清单，减少不必要的 API 调用

### HEARTBEAT.md 示例

```markdown
# HEARTBEAT.md

## 快速检查清单

- [ ] 检查是否有紧急任务
- [ ] 检查子任务状态
- [ ] 无任务时回复 HEARTBEAT_OK

## 定期任务（每小时）

- [ ] 检查系统资源
- [ ] 检查任务队列
```

## 🔍 故障排查

### 如果仍然卡顿

1. **检查系统资源**:
   ```bash
   top
   free -h
   df -h
   ```

2. **检查 Docker 容器**:
   ```bash
   docker compose ps
   docker stats
   ```

3. **减少并发任务数**:
   - 降低批量处理并发数（从 5 降到 3）
   - 减少子任务数量

4. **临时降低心跳频率**:
   - 改为 5 分钟 1 次
   - 观察系统稳定性

### 如果 Token 消耗过高

1. **优化 HEARTBEAT.md**:
   - 保持简洁
   - 避免复杂检查
   - 无任务时快速回复 HEARTBEAT_OK

2. **调整心跳频率**:
   - 改为 2-5 分钟 1 次
   - 根据实际需求平衡
