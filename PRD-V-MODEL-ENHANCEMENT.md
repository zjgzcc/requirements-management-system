## 1. 需求类型与层级

### 1.1 功能描述
支持多种需求类型和父子层级关系，实现需求的逐级分解。

### 1.2 需求类型

| 类型 | 英文 | 说明 | 图标 |
|------|------|------|------|
| 史诗 | Epic | 大型需求，可分解为多个特性 | 🏔️ |
| 特性 | Feature | 功能特性，可分解为多个用户故事 | ✨ |
| 用户故事 | User Story | 用户角度的需求描述 | 👤 |
| 缺陷 | Bug | 需要修复的问题 | 🐛 |

### 1.3 数据结构

```json
{
  "id": "req_xxx",
  "projectId": "proj_xxx",
  "type": "story",
  "parentId": "parent_req_xxx",
  "title": "用户登录功能",
  "description": "富文本内容",
  "priority": "high",
  "severity": "critical",
  "status": "approved",
  "assignee": "user_xxx",
  "reporter": "user_yyy",
  "tags": ["登录", "安全"],
  "customFields": {}
}
```

### 1.4 优先级

| 优先级 | 英文 | 颜色 |
|--------|------|------|
| 紧急 | Urgent | 🔴 |
| 高 | High | 🟠 |
| 中 | Medium | 🟡 |
| 低 | Low | 🟢 |

### 1.5 状态流转

```
草稿 → 评审中 → 已批准 → 开发中 → 已实现 → 已验证 → 已关闭
```

### 1.5 界面设计

**需求树视图**:
```
🏔️ Epic-用户管理系统
  └─ ✨ Feature-登录注册
      ├─ 👤 用户注册
      ├─ 👤 用户登录
      └─ 👤 密码找回
  └─ ✨ Feature-权限管理
      ├─ 👤 角色管理
      └─ 👤 权限分配
```

## 2. 测试计划与执行

### 2.1 功能描述
管理测试计划，记录测试执行结果，跟踪测试进度。

### 2.2 测试计划

**数据结构**:
```json
{
  "id": "plan_xxx",
  "projectId": "proj_xxx",
  "name": "V1.0 系统测试",
  "description": "V1.0 版本的系统测试计划",
  "type": "system", // unit/integration/system/acceptance
  "startDate": "2026-03-20",
  "endDate": "2026-03-30",
  "owner": "user_xxx",
  "testcases": ["tc1", "tc2", "tc3"],
  "status": "in_progress", // draft/active/completed/cancelled
  "progress": 65,
  "createdAt": "2026-03-16",
  "updatedAt": "2026-03-18"
}
```

### 2.3 测试执行

**数据结构**:
```json
{
  "id": "exec_xxx",
  "planId": "plan_xxx",
  "testcaseId": "tc_xxx",
  "executor": "user_xxx",
  "executedAt": "2026-03-21T10:00:00Z",
  "result": "passed", // passed/failed/blocked/skipped
  "actualResult": "实际结果描述",
  "notes": "备注信息",
  "attachments": [],
  "defectId": "bug_xxx" // 如果失败，关联的缺陷
}
```

### 2.4 测试结果

| 结果 | 英文 | 颜色 | 说明 |
|------|------|------|------|
| 通过 | Passed | 🟢 | 测试通过，符合预期 |
| 失败 | Failed | 🔴 | 测试失败，发现缺陷 |
| 阻塞 | Blocked | ⚫ | 无法执行，被阻塞 |
| 跳过 | Skipped | ⚪ | 不执行此用例 |

### 2.5 界面设计

**测试计划列表**:
```
┌────────────────────────────────────────────────┐
```

## 3. V 模型可视化

### 3.1 功能描述
以 V 模型图形化展示需求与测试的对应关系。

### 3.2 V 模型结构

```
                    用户需求 (UR)
                         ↓
                    系统需求 (SR)
                         ↓
                    架构设计 (AD)
                         ↓
                    详细设计 (DD)
                ─────────────────────
                    单元测试 (UT)
                         ↑
                    集成测试 (IT)
                         ↑
                    系统测试 (ST)
                         ↑
                    验收测试 (AT)
```

### 3.3 对应关系

| 开发阶段 | 测试阶段 | 追踪关系 |
|---------|---------|---------|
| 用户需求 | 验收测试 | UR → AT |
| 系统需求 | 系统测试 | SR → ST |
| 架构设计 | 集成测试 | AD → IT |
| 详细设计 | 单元测试 | DD → UT |

## 4. 影响分析

### 4.1 功能描述
分析需求变更的影响范围，识别需要重新测试的项目。

### 4.2 影响范围

**直接影响**:
- 直接关联的测试用例
- 直接关联的缺陷

**间接影响**:
- 子需求
- 兄弟需求
- 父需求的其它子需求

**测试计划影响**:
- 包含相关用例的测试计划
- 需要重新执行的测试

### 4.3 API 设计

**请求**:
```
GET /api/requirements/:id/impact-analysis
```

**响应**:
```json
{
  "success": true,
  "data": {
    "requirement": {
      "id": "req_xxx",
      "title": "用户登录功能",
      "changeDescription": "修改了密码验证逻辑"
    },
    "directImpacts": [
      {
        "type": "testcase",
        "id": "tc_001",
        "title": "登录功能测试",
        "status": "passed"
      },
      {
        "type": "testcase",
        "id": "tc_002",
        "title": "密码强度测试",
        "status": "not_executed"
      }
    ],
    "indirectImpacts": [
      {
        "type": "requirement",
        "id": "req_child",
        "title": "密码找回功能",
        "relation": "child"
      }
    ],
    "affectedTestPlans": [
      {
        "id": "plan_001",
        "name": "V1.0 系统测试",
        "status": "in_progress"
      }
    ],
    "summary": {
      "totalTestcases": 5,
      "needRetest": 3,
      "affectedPlans": 2
    }
  }
}
```

## 5. 项目仪表盘

### 5.1 功能描述
可视化展示项目进度、质量、团队效能等关键指标。

### 5.2 核心指标

**需求指标**:
- 需求总数
- 已完成需求
- 进行中需求
- 需求变更率
- 需求稳定性

**测试指标**:
- 测试用例总数
- 已执行用例
- 通过率
- 缺陷密度
- 测试覆盖率

**进度指标**:
- 迭代进度
- 燃尽图
- 累计流图

### 5.3 图表类型

**饼图** - 需求状态分布:
```
     需求状态分布
   ╭─────────────╮
   │ 已完成 60%  │
   │ 进行中 30%  │
   │ 待开始 10%  │
   ╰─────────────╯
```

**柱状图** - 每日新增/完成:
```
数量
  │
10│     ████
  │  ████████
  │██████████
  └───────────→ 日期
```

**燃尽图** - 迭代进度:
```
剩余工作量
  │\
  │ \
  │  \
  │   \____
  └────────→ 时间
```

### 5.4 界面布局

```

│ ┌─────────────────┐ ┌─────────────────┐       │
│ │  需求状态分布   │ │   迭代燃尽图    │       │
│ │    [饼图]       │ │    [折线图]     │       │
│ └─────────────────┘ └─────────────────┘       │
├────────────────────────────────────────────────┤
```
