# 医疗器械需求管理系统合规调研报告

**调研日期：** 2026 年 3 月 14 日  
**调研对象：** 医疗器械需求管理系统合规要求  
**适用法规：** ISO 13485、FDA 21 CFR Part 820、21 CFR Part 11

---

## 一、ISO 13485 对需求管理的要求

### 法规名称
ISO 13485:2016《医疗器械 质量管理体系 用于法规的要求》

### 具体要求条款

| 条款号 | 条款名称 | 具体要求 |
|--------|----------|----------|
| 7.3.2 | 设计和开发策划 | 应确定设计和开发阶段、评审、验证和确认活动；明确职责和权限；管理接口确保沟通 |
| 7.3.3 | 设计和开发输入 | 应确定与产品要求相关的输入，包括功能、性能、安全、法规要求；输入应完整、清楚、不自相矛盾 |
| 7.3.4 | 设计和开发输出 | 输出应满足输入要求；包含或引用产品接收准则；规定对安全正常使用至关重要的产品特性 |
| 7.3.5 | 设计和开发评审 | 在适当阶段进行系统评审，评价结果满足要求的能力；识别问题并提出必要措施 |
| 7.3.6 | 设计和开发验证 | 通过提供客观证据，验证输出满足输入要求 |
| 7.3.7 | 设计和开发确认 | 通过提供客观证据，确认产品满足规定的使用要求或预期用途 |
| 7.3.8 | 设计和开发转换 | 确保设计和开发输出在成为最终产品规范前得到验证和确认 |
| 7.3.9 | 设计和开发更改的控制 | 识别、记录、评审、验证、确认并批准更改；评估对组成部分和已交付产品的影响 |
| 7.5.9 | 可追溯性 | 建立可追溯性程序，确保能够追溯产品实现的全过程 |

### 技术实现建议

1. **需求管理流程**
   - 建立结构化的需求捕获模板，包含需求 ID、描述、来源、优先级、验收标准
   - 实施需求评审工作流，支持多角色会签
   - 需求变更需经过影响分析和批准流程

2. **需求追溯**
   - 建立双向追溯链接：需求→设计→实现→验证→确认
   - 使用唯一标识符追踪每个需求
   - 实现追溯矩阵的自动生成和可视化

3. **文档控制**
   - 版本控制系统记录所有需求文档变更
   - 电子签名支持评审和批准流程
   - 保留历史版本供审计查阅

### 合规检查清单

- [ ] 需求输入是否完整、清晰、无矛盾
- [ ] 是否建立了需求评审流程并执行
- [ ] 需求是否可追溯到用户需求和法规要求
- [ ] 设计输出是否满足需求输入
- [ ] 是否进行了需求验证和确认
- [ ] 需求变更是否有记录和批准
- [ ] 是否建立了需求追溯矩阵
- [ ] 相关人员是否经过培训

## 二、FDA 21 CFR Part 820 相关要求

### 法规名称
21 CFR Part 820《质量管理体系法规》(QMSR)

### 具体要求条款

| 条款号 | 条款名称 | 具体要求 |
|--------|----------|----------|
| §820.10 | 质量管理体系要求 | 必须建立符合 ISO 13485 的质量管理体系文档 |
| §820.30 | 设计控制 | 适用于 II 类、III 类和部分 I 类医疗器械；必须建立设计控制程序 |
| §820.30(b) | 设计和开发策划 | 制定计划，描述设计开发活动，明确职责、接口、资源 |
| §820.30(c) | 设计输入 | 确定适当的产品要求，包括预期用途、用户需求；解决不完整、模糊或矛盾的要求 |
| §820.30(d) | 设计输出 | 以能够对照设计输入进行验证的方式记录；包含验收标准；确定关键产品特性 |
| §820.30(e) | 设计评审 | 记录评审结果和结论；包括识别问题及必要措施 |
| §820.30(f) | 设计验证 | 通过检查和提供客观证据，确认输出满足输入要求 |
| §820.30(g) | 设计确认 | 在规定的操作条件下，确认设备符合用户需求和预期用途 |
| §820.30(h) | 设计转换 | 确保设计输出正确转换为生产规范 |
| §820.30(i) | 设计变更 | 建立变更识别、记录、验证/确认、批准的程序 |
| §820.30(j) | 设计历史文件 (DHF) | 为每种器械类型建立 DHF，包含或引用记录以证明符合设计控制要求 |

### 技术实现建议

1. **设计控制工作流**
   - 实现阶段门控 (Stage-Gate) 流程，每个阶段需完成评审才能进入下一阶段
   - 自动检查设计输入输出的一致性
   - 支持并行开发和迭代开发模式

2. **设计历史文件 (DHF)**
   - 电子化 DHF 管理，自动归档设计文档
   - 支持按产品类型组织设计记录
   - 实现快速检索和审计准备

3. **风险管理集成**
   - 与 ISO 14971 风险管理流程集成
   - 需求关联风险分析和控制措施
   - 设计评审包含风险评估结果

### 合规检查清单

- [ ] 是否建立了设计控制程序
- [ ] 设计计划是否明确职责和接口
- [ ] 设计输入是否文档化并评审
- [ ] 设计输出是否符合输入要求
- [ ] 是否进行了设计评审并记录
- [ ] 设计验证和确认是否完成
- [ ] 是否建立了设计历史文件 (DHF)
- [ ] 设计变更是否经过批准
- [ ] 相关人员是否具备相应资质

## 三、21 CFR Part 11 电子签名要求

### 法规名称
21 CFR Part 11《电子记录；电子签名》

### 具体要求条款

| 条款号 | 条款名称 | 具体要求 |
|--------|----------|----------|
| §11.1 | 范围 | 规定电子记录和电子签名被视为可信、可靠且等同于纸质记录和手写签名的标准 |
| §11.10 | 封闭系统控制 | 必须采用程序和控制确保电子记录的真实性、完整性、保密性 |
| §11.10(a) | 系统验证 | 验证系统以确保准确性、可靠性、一致的性能，能够识别无效或篡改的记录 |
| §11.10(d) | 访问控制 | 限制系统访问仅限授权人员 |
| §11.10(e) | 审计日志 | 使用安全、计算机生成、带时间戳的审计日志，独立记录操作员创建、修改、删除电子记录的日期和时间 |
| §11.10(g) | 权限检查 | 确保只有授权人员可以使用系统、电子签名记录、访问系统、修改记录 |
| §11.50 | 签名表现 | 签名记录必须包含：(1) 签名者打印姓名；(2) 签名日期和时间；(3) 签名含义（如审核、批准、责任或作者） |
| §11.70 | 签名/记录链接 | 电子签名必须与相应电子记录链接，确保签名不能被切除、复制或转移以伪造记录 |
| §11.100 | 电子签名一般要求 | 电子签名必须是唯一的，不能由他人重用或重新分配 |
| §11.200 | 电子签名组件和控制 | 签名至少包含两个组件：用户名和密码；或生物识别等 |

### 技术实现建议

1. **电子签名实现**
   - 双因素认证：用户名/密码 + 二次确认
   - 签名时强制重新输入密码或进行生物识别
   - 签名后显示签名者姓名、时间戳、签名含义
   - 签名与记录加密绑定，防止篡改

2. **用户身份管理**
   - 唯一用户标识，禁止共享账户
   - 强密码策略：最小长度 8 位，包含大小写字母、数字、特殊字符
   - 密码定期更换（如 90 天）
   - 账户锁定策略：连续失败 5 次后锁定

3. **签名法律效力**
   - 首次使用电子签名前，用户需签署声明确认其法律效力
   - 记录签名者的身份认证方式
   - 签名不可撤销，但可添加注释说明

### 合规检查清单

- [ ] 电子签名是否唯一且不可重用
- [ ] 签名是否包含姓名、时间、含义
- [ ] 签名是否与记录安全链接
- [ ] 是否实施用户身份验证
- [ ] 密码策略是否符合要求
- [ ] 用户是否确认电子签名的法律约束力
- [ ] 签名记录是否可追溯
- [ ] 系统是否经过验证

## 四、需求追溯矩阵 (RTM) 的标准格式

### 法规名称
ISO 13485:2016 Clause 7.5.9 / 21 CFR Part 820.30

### 具体要求

需求追溯矩阵 (Requirements Traceability Matrix, RTM) 是证明需求从来源到验证全生命周期可追溯的关键文档。

### 标准 RTM 格式

| 需求 ID | 需求描述 | 来源/类型 | 优先级 | 设计元素 ID | 验证测试 ID | 确认活动 ID | 状态 | 备注 |
|---------|----------|-----------|--------|-------------|-------------|-------------|------|------|
| REQ-001 | 设备应能在 5-40°C 环境温度下工作 | 用户需求 | 高 | DES-001 | TEST-001 | VAL-001 | 已验证 | - |
| REQ-002 | 电池续航时间≥8 小时 | 法规要求 (IEC 60601-1) | 高 | DES-002 | TEST-002 | VAL-002 | 已验证 | - |
| REQ-003 | 用户界面响应时间<2 秒 | 用户需求 | 中 | DES-003 | TEST-003 | - | 验证中 | - |

### 追溯类型

1. **前向追溯 (Forward Traceability)**
   - 需求 → 设计 → 实现 → 测试
   - 确保每个需求都被实现和验证

2. **后向追溯 (Backward Traceability)**
   - 测试 → 实现 → 设计 → 需求
   - 确保每个设计元素和测试都有明确的需求依据

3. **双向追溯 (Bidirectional Traceability)**
   - 同时实现前向和后向追溯
   - FDA 和 ISO 推荐的最佳实践

### 技术实现建议

1. **自动化追溯**
   - 使用需求管理工具自动维护追溯链接
   - 变更影响分析自动识别受影响的下游元素
   - 追溯覆盖率自动计算和报告

2. **追溯可视化**
   - 图形化追溯关系图
   - 追溯缺口高亮显示
   - 状态仪表板实时显示追溯完整性

3. **报告功能**
   - 按需生成 RTM 报告（PDF/Excel）
   - 支持按需求类型、状态、优先级筛选
   - 审计追踪追溯变更历史

### 合规检查清单

- [ ] 是否建立了 RTM 模板
- [ ] 每个需求是否有唯一标识符
- [ ] 需求是否追溯到来源（用户需求/法规）
- [ ] 每个需求是否有对应的设计元素
- [ ] 每个需求是否有验证测试
- [ ] 关键需求是否有确认活动
- [ ] RTM 是否定期评审和更新
- [ ] 追溯缺口是否被识别和处理
- [ ] RTM 是否作为 DHF 的一部分保存

## 五、审计日志 (Audit Trail) 要求

### 法规名称
21 CFR Part 11 §11.10(e)

### 具体要求条款

| 要求项 | 具体内容 |
|--------|----------|
| 独立性 | 审计日志必须独立记录，与业务数据分离存储 |
| 时间戳 | 所有操作必须记录准确的日期和时间，时间同步且不可篡改 |
| 完整性 | 记录创建、修改、删除电子记录的所有操作员操作 |
| 不可篡改 | 已记录的审计日志不能被修改或删除 |
| 可读性 | 审计日志应能以人类可读形式呈现 |
| 保留期 | 审计日志保留期限至少与相应电子记录相同 |
| 可审查性 | 审计日志应可供 FDA 审查和复制 |

### 审计日志应记录的信息

| 字段 | 说明 | 示例 |
|------|------|------|
| 时间戳 | 操作发生的日期和时间（精确到秒） | 2026-03-14 10:30:45 |
| 用户 ID | 执行操作的用户唯一标识 | user_12345 |
| 用户姓名 | 执行操作的用户全名 | 张三 |
| 操作类型 | 创建/修改/删除/查看/导出/签名 | 修改 |
| 对象类型 | 被操作的对象类型 | 需求文档 |
| 对象 ID | 被操作对象的唯一标识 | REQ-001 |
| 修改前值 | 修改前的数据内容（如适用） | 优先级：中 |
| 修改后值 | 修改后的数据内容（如适用） | 优先级：高 |
| 修改原因 | 变更原因或注释 | 根据客户反馈调整优先级 |
| 客户端信息 | 操作来源的 IP 地址/设备信息 | 192.168.1.100 / Chrome 120 |

### 技术实现建议

1. **日志架构**
   - 使用只追加 (Append-Only) 日志存储
   - 日志数据与业务数据物理分离
   - 实施日志完整性校验（如哈希链）
   - 考虑使用区块链或 WORM（一次写入多次读取）存储

2. **日志保护**
   - 日志访问权限严格控制，仅审计员可访问
   - 日志传输和存储加密
   - 定期备份日志到安全位置
   - 实施日志监控和异常告警

3. **日志查询**
   - 提供灵活的日志查询和过滤功能
   - 支持按时间范围、用户、操作类型、对象筛选
   - 可导出标准格式（CSV、PDF）供审计使用
   - 实现预定义审计报告模板

4. **时间同步**
   - 使用 NTP 协议同步服务器时间
   - 记录时区信息
   - 防止用户修改本地时间影响日志

### 合规检查清单

- [ ] 审计日志是否自动记录所有关键操作
- [ ] 日志是否包含完整的时间戳和用户信息
- [ ] 日志是否不可篡改和删除
- [ ] 修改操作是否记录前后值
- [ ] 日志保留期限是否符合要求
- [ ] 日志是否可供审查和导出
- [ ] 是否有日志访问控制
- [ ] 系统时间是否同步和受保护
- [ ] 是否定期审查审计日志

## 六、数据加密和访问控制要求

### 法规名称
21 CFR Part 11 §11.10 / §11.30；ISO 13485 Clause 7.5.3

### 具体要求条款

| 条款 | 要求内容 |
|------|----------|
| §11.10(d) | 限制系统访问仅限授权人员 |
| §11.10(g) | 使用权限检查确保只有授权人员可以访问和操作 |
| §11.30 | 开放系统需采用文档加密和数字签名标准确保记录的真实性、完整性和保密性 |
| ISO 13485 7.5.3 | 在产品实现的全过程中，使用适当的方法识别产品；在有可追溯性要求的场合，控制并记录产品的唯一性标识 |

### 数据加密要求

#### 传输加密
| 要求 | 技术实现 |
|------|----------|
| 通信协议 | 强制使用 TLS 1.2 或更高版本 |
| 证书管理 | 使用可信 CA 签发的 SSL/TLS 证书 |
| 加密强度 | 最低 AES-128，推荐 AES-256 |
| API 安全 | REST API 使用 HTTPS，实施 API 密钥或 OAuth 认证 |

#### 存储加密
| 要求 | 技术实现 |
|------|----------|
| 数据库加密 | 透明数据加密 (TDE) 或应用层加密 |
| 文件加密 | 敏感文件使用 AES-256 加密存储 |
| 密钥管理 | 使用 HSM 或云 KMS 管理加密密钥 |
| 密钥轮换 | 定期轮换加密密钥（如每年） |
| 备份加密 | 备份数据必须加密存储 |

### 访问控制要求

#### 身份认证
| 要求 | 技术实现 |
|------|----------|
| 唯一身份 | 每个用户有唯一账户，禁止共享 |
| 强密码策略 | 最小 8 位，含大小写字母、数字、特殊字符 |
| 多因素认证 | 关键操作强制 MFA（密码 + 短信/令牌/生物识别） |
| 会话管理 | 空闲超时自动登出（如 15 分钟） |
| 账户锁定 | 连续失败 5 次后锁定，需管理员解锁 |

#### 授权管理
| 要求 | 技术实现 |
|------|----------|
| 角色基础访问控制 (RBAC) | 定义角色（如管理员、审核员、普通用户） |
| 最小权限原则 | 用户仅获得完成工作所需的最小权限 |
| 权限分离 | 关键操作需多人授权（如四人眼原则） |
| 权限审查 | 定期审查用户权限（如每季度） |
| 离职处理 | 员工离职立即禁用账户 |

#### 访问审计
| 要求 | 技术实现 |
|------|----------|
| 登录日志 | 记录所有登录尝试（成功/失败） |
| 访问日志 | 记录敏感数据访问 |
| 异常检测 | 监控异常访问模式并告警 |
| 定期报告 | 生成访问控制合规报告 |

### 技术实现建议

1. **加密架构**
   ```
   用户 → HTTPS(TLS 1.3) → 应用服务器 → 加密连接 → 数据库
                                    ↓
                              KMS/HSM(密钥管理)
   ```

2. **访问控制模型**
   - 实现基于角色的访问控制 (RBAC)
   - 支持基于属性的访问控制 (ABAC) 用于细粒度控制
   - 实施数据级权限（行级/列级安全）

3. **安全开发生命周期**
   - 安全需求纳入设计阶段
   - 代码安全审查
   - 渗透测试和漏洞扫描
   - 安全补丁管理

### 合规检查清单

- [ ] 数据传输是否使用 TLS 加密
- [ ] 敏感数据是否加密存储
- [ ] 加密密钥是否安全管理
- [ ] 是否实施唯一用户身份
- [ ] 密码策略是否符合要求
- [ ] 是否实施多因素认证
- [ ] 是否基于角色分配权限
- [ ] 是否遵循最小权限原则
